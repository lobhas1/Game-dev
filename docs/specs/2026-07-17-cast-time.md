# Cast-time round — real cast windows in the recorded timeline

## Ratified decisions (design review, 2026-07-16)

1. **Declared bands are binding.** The corpus's own `cast.castTime` bands (43/61 spells declare
   `mode:"cast"` with a band) define the windows. No cost/tier-derived mapping — the axis is already
   priced and declared; this round makes it *visible*.
2. **Instants untouched.** The 18 `mode:"instant"` spells keep a 0-second window; their 15% budget
   tax (`CastMultiplier` 0.85) remains the price of instancy. No pricing value moves.
3. **Depth (a) — timeline-only.** The world clock already ticks through windups
   (`Sim.Cast` step 2, CombatSim.cs); fights are already the right length. This round repairs the
   *recording* of time, not the simulation of it. Interactive windups (opponent decisions
   mid-windup, interrupts, `interruptible`/`moveWhileCasting` consumption) stay out of scope.

## The defect being repaired

The sim ticks real windups, but replays collapse them: `ReplayRecorder.Mark` stamps every event
accumulated since the last mark with the *current* clock, and both call sites mark only after
`Cast()` returns — so `CastStarted` (emitted pre-windup) lands on the effect instant. Committed
evidence: `beacon-582c9048` (cast/quick) records CastStarted AND its damage both at t=0.400;
`lighthouse-01d420e1` (cast/normal) both at t=0.800. No replay anywhere shows a CastStarted→effect
gap; the renderer's cast-timing workarounds are downstream of exactly this.

## The fix (two parts)

### 1. Emission-time stamping — the principled repair of Mark's batching

`EventBus.Emit` stamps each event with `State.Now` at **emission** (the bus gets a clock delegate
from `Sim`); `ReplayRecorder.Mark` consumes the per-event emission times instead of flattening the
batch to the mark instant. Mark's call sites do not move — batching disappears as a *timestamping*
concern while remaining a harvesting cadence. Zone ticks and DoTs that fire during someone's windup
also get true times for free.

### 2. `CastResolved` — the renderer's resolution anchor

A new sim event, emitted **once per successful cast, immediately after the windup completes and
before delivery/effects**:

- record: `CastResolved(EntityRef Caster, SpellId Spell)`; canonical line `castResolved E<n> <spell>`.
- Event order per successful cast: `CastStarted` → *(windup; world ticks)* → `CastResolved` →
  delivery/effects. For instants, `CastResolved` shares `CastStarted`'s timestamp.
- Failed casts keep today's shape — `CastStarted` → `CastFailed` — and **never** emit `CastResolved`.

### Renderer-contract addition (flag for the Unreal resync)

The replay schema stays `"1"` — this is an **additive** event type, same envelope
(`t`/`type`/`canonical`/`payload`), payload `{caster, spell}` mirroring `CastStarted`. Contract:
**the cast bar runs from `CastStarted.t` to `CastResolved.t`; effect VFX anchor at `CastResolved`.**
The Unreal renderer needs a `CastResolved` handler at resync; until then unknown-type events may be
skipped (the bar falls back to zero-width, today's behavior).

## The binding window table (BalanceTables.CastTimeBands — values unchanged, now visible)

| band | seconds | corpus usage (61) |

"Exact" for criterion (1) means the recorded gap equals the compiled `CastTime` within 1e-3 (the
clock accumulates in 0.05s float quanta; e.g. 0.4 records as 0.40000004).

## Pre-registered criteria (frozen)

1. **Band ↔ gap, all 61 showcases:** per spell, `t(CastResolved) − t(CastStarted)` equals the
   declared band's seconds exactly (tolerance above), and the first effect event carries
   `CastResolved`'s timestamp (no delayed templates exist in the canon).
2. **Fight-outcome invariance:** same winner, same final HP, same event multiset modulo timestamps
   and the new `castResolved` lines — all 3 M1 replays and the tournament. (`results.csv` is
   expected byte-identical: no state, RNG draw, or metric reads the new event — `VerbOf` defaults
   null, so C3's distinct-verb count cannot move.)
3. **Phase A C1–C4 re-run offline** on the v4 kits at the registered thresholds (C1 ≥70%; C2 median
   15–90s; C3 ≥3 verbs/fight; C4 ≥50% swings), config hpScale 8 / mana 250 / amplifyMajor 1.75,
   seeds 1..5. **Expected unchanged** — the world clock already ticked through windups, so durations
   cannot move. Any criterion failing → STOP, file the FAIL doc, no silent retuning.
4. **Golden re-blessed:** the frost-ember seed-1 projection gains `castResolved` lines (one per
   successful cast). The old golden is archived (never overwritten); the new projection is blessed
   and `V3Tests.Defaults_ReproduceTodaysFight_Seed1` updated against it.
5. **Diff report:** across the 3 M1 replays and all 61 showcases, the only changes are (a) event
   timestamps and (b) added `castResolved` lines. Any effect amount, order inversion (beyond the
   insertion), or vanished line is a FAIL.

## Immutables

`prompts/**` (all three shas), `fixtures/seeds/**`, `fixtures/anchors/**`, `arena/fusions/*.record.json`,
all experiment docs, all registered thresholds. Also explicitly unmoved: `CastTimeBands` seconds,
`CastMultiplier` pricing, cooldowns, costs — no balance value changes this round.

## Out of scope (banked)

Interactive windups — opponent decisions during a cast, interrupt rules (`interruptible` parses but
is consumed nowhere), `moveWhileCasting`, cast-bar UI semantics beyond the two anchor events. The
`long` band still has no corpus material (tracked by the coverage doc's missing-tokens list).
